## 1. The productisation problem

Every decentralised compute subnet has a version of the same problem: it produces outputs that
are plausibly correct but not *provably* correct. The outputs come from nodes the operator does
not control, running code the subnet cannot verify, producing results the validator can sample
but not fully audit.

For internal use — experimentation, research, R&D benchmarks — this is fine. "Plausibly correct"
is good enough. But for selling the outputs to external buyers, "plausibly correct" is not enough.

An enterprise buyer — a financial firm, a healthcare provider, a regulated AI company — cannot
sign a service agreement for outputs that might be wrong, generated by code running in an
environment the service provider cannot audit, scored by validators who could in principle be
bribed. They need SLAs. SLAs require accountability. Accountability requires that you can *prove*
what happened.

This is the productisation gap. Non-TEE subnets fill it with trust in the team running the subnet.
TEE subnets close it with hardware proofs that do not require trusting any individual party.

## 2. What non-TEE subnet outputs can and cannot prove

| Claim | Can non-TEE prove it? | How the claim is established |
|---|---|---|
| "Our node ran model X" | No | Miner self-reports; validator spot-checks a sample |
| "No outputs were fabricated" | No | Statistical — enough sampling makes fabrication unlikely |
| "The scoring was fair" | No | Trust in validators; stake-weighted averaging limits manipulation |
| "The code was unmodified" | No | Deployment trust in node operators |
| "No one read the inputs" | No | Transport encryption, but compute is in cleartext |
| "This output came from node N" | Approximately | Cryptographic signing, but the key is generated in unattested code |

This is not unique to any specific subnet — it is a fundamental property of compute subnets
without hardware attestation. The outputs are as trustworthy as the people running the nodes.

For many use cases this is acceptable. For productisable outputs sold to regulated buyers,
or outputs used as oracle inputs to on-chain smart contracts, or inference results where model
IP needs protection, it is not.

## 3. What TEE adds: a verifiable quality floor

The TEE subnet changes the answer to every question in the table above:

| Claim | Can TEE prove it? | How the proof works |
|---|---|---|
| "Our node ran model X" | ✓ | EXPECTED_MEASUREMENT — measurement mismatch → score=0.0 (`subnet/tee/verifier.py` step 6) |
| "No outputs were fabricated" | ✓ | RA-TLS session key — only the attested enclave can sign outputs (`subnet/tee/ratls/session.py`) |
| "The scoring was fair" | ✓ (with 66% attestation) | Requires majority stake collusion to inflate; collusion is slashable |
| "The code was unmodified" | ✓ | Measurement = SHA-384 of the initial enclave image (`subnet/tee/backends/tdx.py`) |
| "No one read the inputs" | ✓ | RA-TLS WorkEnvelope AES-256-GCM — operator cannot read enclave memory |
| "This output came from node N" | ✓ | Identity binding: sha256(peer_id:epoch) in hardware-signed quote (`subnet/tee/quote.py`) |

The proof is not trust in a team. It is a cryptographic certificate from the CPU — hardware that
the operator, the cloud provider, and the subnet owner cannot fake.

This is what "verifiable quality floor" means: the minimum acceptable output quality is enforced
by hardware, not by the good behaviour of node operators.

## 4. The four business-critical properties

### 4.1 Predictable quality floor

Without TEE, the quality of subnet outputs depends on the mix of honest vs. dishonest miners
at any given time. In a competitive subnet, the equilibrium tends toward the minimum effort that
passes validation sampling. The quality floor is the sampling rate, not the actual task quality.

With TEE + `EXPECTED_MEASUREMENT`, every miner earning emissions is provably running the exact
same binary. The quality floor is enforced by the code, not by the economics of getting caught.

**What this enables:** SLA commitments. A business can commit to customers that outputs are
produced by verified code, not just "probably the right code." When the measurement is locked,
a new binary version requires a subnet governance action — not an individual node operator's
decision.

### 4.2 Miner accountability

Without TEE, a miner can claim to run an expensive model and run a cheaper one. The only evidence
is the output — and outputs from a lighter model may pass spot-checks.

With TEE, a miner cannot earn emissions by running a different binary. There is no cheaper
imposter path: either you run the correct binary in genuine TEE hardware, or you earn 0. This
makes honest compute the economically rational strategy — not by moral argument, but by making
dishonest compute economically non-viable.

**What this enables:** Predictable cost structures for subnet operators. If you know all earning
nodes are running the same code, you can predict output quality, latency, and resource usage.
You can price your service accordingly.

### 4.3 Verifiable SLA commitments

A traditional SLA says "we will deliver X quality with Y uptime." Proving the SLA was met
requires trusting the service provider's logs.

A TEE subnet SLA is different. The attestation quote for each epoch's output is a hardware-signed
timestamp proving:
- Which binary ran
- Which node it ran on
- Which epoch it ran in
- That no one read the inputs or modified the outputs in transit

These proofs are on the chain (as attestation submissions) and in the DHT (as DCAP quotes).
They are independently verifiable by any third party with access to Intel's or AMD's public
root CA — without trusting the subnet operator's claim.

**What this enables:** Audit trails that survive disputes. If a customer claims the subnet
delivered incorrect output, the on-chain attestation record shows whether the correct binary
was running on that epoch. This is not a log that a service provider can edit.

### 4.4 Model IP protection

For subnets where the value is in the model itself — a fine-tuned LLM, a proprietary scoring
model, a specialised embedding model — the model weights are the business. Deploying them on
a network of nodes controlled by third parties means accepting that those operators can read
the model weights, copy them, and compete with you.

In a TEE enclave, model weights loaded into the enclave are:
- Encrypted at the hardware level (operator cannot read them from memory)
- Sealed with the enclave's measurement (a different binary version cannot load them)
- Transmitted to validators over RA-TLS (encrypted channel that only the enclave can decrypt)

**What this enables:** Deploying proprietary models on a decentralised network without giving
operators access to the weights. The subnet owner publishes the binary (whose measurement is
locked by EXPECTED_MEASUREMENT); operators run it; neither the operators nor any network
participant can extract the weights from the running enclave.

## 5. Use cases

### 5.1 Verifiable inference for regulated industries

**The problem:** A healthcare company wants to use decentralised LLM inference for clinical
decision support. They need to prove to regulators that a specific model version was used, that
inputs were not readable by node operators, and that outputs were not modified in transit.

**How TEE solves it:** The measurement lock proves the model version. RA-TLS WorkEnvelope proves
input confidentiality. The OutputEnvelope HMAC proves output integrity. All three proofs are
hardware-rooted and independently verifiable.

**Non-TEE alternative:** Trust in the subnet operator's claims. Not acceptable for regulated use.

### 5.2 On-chain oracle from verifiable compute

**The problem:** An on-chain smart contract needs risk scores for financial addresses. It cannot
accept scores from an unverified off-chain source — it needs proof that the scores were produced
by a specific algorithm, not fabricated.

**How TEE solves it:** The oracle includes the DCAP attestation alongside each score batch. Smart
contract verifiers can check the attestation against Intel's root CA. If the measurement matches
the published algorithm hash, the score is provably from the correct code.

**Non-TEE alternative:** Trust in the oracle operator. On-chain DeFi protocols historically do not
accept this trust model for high-value oracles.

### 5.3 Proprietary model subnet — monetise without exposure

**The problem:** A company has trained a proprietary embedding model. They want to offer it as a
subnet service — other protocols pay to query it — without exposing the weights to the node
operators running the inference.

**How TEE solves it:** Model weights are loaded into the enclave at startup and sealed. Operators
never see the weights. Customers query via RA-TLS (encrypted channel). The measurement lock
ensures the published binary (which includes the sealed weight loading code) is what runs.

**Non-TEE alternative:** Host the model yourself (centralised, no subnet economics) or accept
that all node operators can copy your model.

### 5.4 Verifiable AI training contribution

**The problem:** Miners contributing to a distributed training run want to prove their contribution
was included. The training coordinator wants to prove that all miners ran the correct model
version and did not inject corrupted gradients.

**How TEE solves it:** Each training step generates a DCAP quote bound to the epoch. The
measurement verifies the training code. RA-TLS encrypts gradient exchanges. Corrupted gradients
from modified code would produce a different measurement, earning 0 instead of contributing.

## 6. The cost/benefit for subnet owners

### What TEE costs

**Hardware:** Miners need TDX or SEV-SNP capable hardware. Cloud options (Azure DCasv5, GCP N2D)
are generally available. Bare metal Intel Sapphire Rapids or AMD EPYC Genoa. Cost premium over
non-TEE GPU is roughly 20–40% depending on cloud provider and configuration.

**Operational:** Subnet owners need to:
- Publish and maintain EXPECTED_MEASUREMENT as binaries evolve
- Manage the Gramine manifest for production TDX deployments (see GRAMINE.md)
- Coordinate measurement updates as a subnet governance action (not a silent binary push)

**Complexity:** The TEE template handles all the plumbing — backends, DcapVerifier, RA-TLS,
sealed storage. A subnet developer implements `protocol.py` and `scoring.py` — the same as
any other template fork. The TEE layer is transparent in development with MOCK_TEE=true.

### What TEE saves

**Sampling cost:** Without TEE, validators must sample miner outputs at a high rate to detect
cheating. With TEE, sampling is still valuable for correctness checking (not for fraud detection).
The hardware proof handles fraud detection.

**Support cost:** Output disputes ("did the node run the correct model?") are resolved by checking
the on-chain attestation record. No forensics required.

**Trust overhead:** Enterprise customers typically require extensive due diligence before trusting
a new compute provider. TEE proofs reduce this overhead because the proof is hardware-rooted, not
organisation-rooted.

### The break-even calculation

The TEE overhead is worth accepting when the value of *provable* correctness exceeds the value of
*probable* correctness. This is use-case specific, but a heuristic:

- **TEE makes sense** when outputs are used in regulated decisions, as oracle inputs, or when
  model IP protection is a business requirement
- **TEE makes sense** when the alternative is a centralised service (then TEE adds decentralisation
  without sacrificing verifiability)
- **TEE makes less sense** for purely experimental subnets, internal tooling, or cases where
  statistical correctness verification is sufficient

## 7. What sustainable looks like

A non-TEE subnet's economic stability depends on the ratio of honest to dishonest miners. As the
subnet grows and emissions grow, the incentive to cheat increases. Defences scale with sampling
rate. This is a maintenance race: the subnet team must continuously improve sampling to stay ahead
of increasingly sophisticated cheating.

A TEE subnet's economic stability does not depend on the fraction of honest miners. Every miner
earning emissions is, by definition, running the correct code in genuine hardware. The subnet
owner does not need to improve defences as the subnet grows — the measurement lock and 66%
attestation handle it.

This changes the operational model: instead of "continuously defend against a growing attack
surface," the model is "publish the binary, set the measurement, maintain the chain deployment."
The security work is front-loaded (getting the binary right) rather than ongoing (catching
increasingly sophisticated cheaters).

For a business built on subnet outputs, this matters: the outputs are as reliable at 1000 miners
as they were at 10. The quality floor does not erode with scale.
