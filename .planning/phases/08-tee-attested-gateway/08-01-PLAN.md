# Phase 8 Plan 01: TEE-Attested Gateway

## Objective

Add TEE attestation endpoints and RA-TLS inference forwarding to the Frontier FastAPI gateway. The gateway exposes `/attestation` for on-demand TEE quote retrieval, and upgrades `/v1/chat/completions` to forward requests through RA-TLS verified channels to miner nodes.

## Context

- Frontier app: `subnet/frontier/app.py` (FastAPI, capacity routing, returns 501 stub)
- TEE backends: `subnet/tee/backends/` (MockBackend for dev/test)
- RA-TLS client: `subnet/tee/ratls/client.py` (verifies miner certs)
- RA-TLS server: `subnet/tee/ratls/server.py` (generates attested certs)
- TeePublisher: `subnet/tee/publisher.py` (generates per-epoch quotes)
- Capacity table: `subnet/frontier/capacity.py` (node routing)

## Tasks

### Task 1: Add /attestation endpoint to Frontier app
type="auto"

Add a `GET /attestation` endpoint that returns the gateway's own TEE attestation quote. This proves to callers that the gateway itself runs in a TEE.

**Implementation:**
- Extend `create_app()` to accept an optional `tee_backend: TeeBackendBase` and `gateway_peer_id: str` and `epoch_fn: Callable[[], int]`
- Add `GET /attestation` endpoint (unauthenticated) that generates a fresh quote using the backend and returns the quote as JSON
- Return 503 if no TEE backend is configured
- Include backend type, measurement, timestamp, peer_id in response

**Done criteria:**
- /attestation returns quote JSON with mock backend
- /attestation returns 503 when no backend configured
- Existing endpoints unchanged

### Task 2: Add RA-TLS inference forwarder
type="auto"

Create `subnet/frontier/forwarder.py` — an async HTTP client that forwards inference requests to miner nodes via RA-TLS verified connections.

**Implementation:**
- `RaTlsForwarder` class that wraps httpx.AsyncClient
- `forward(node: NodeEntry, request: ChatCompletionRequest) -> dict` method
- Verifies miner's RA-TLS cert before forwarding (using RaTlsClient)
- Returns parsed JSON response from the miner
- Timeout handling (30s default)
- On verification failure: raises `ForwardingError` with reason

**Done criteria:**
- Forwarder class created with forward method
- Verification failure raises ForwardingError
- Timeout handling present

### Task 3: Wire forwarder into chat completions endpoint
type="auto"

Replace the 501 stub in `/v1/chat/completions` with actual RA-TLS forwarding through the forwarder.

**Implementation:**
- Extend `create_app()` to accept optional `forwarder: RaTlsForwarder`
- When forwarder is provided, call `forwarder.forward(node, request)` instead of returning 501
- On ForwardingError: return 502 with error details
- On timeout: return 504
- When forwarder is None: keep existing 501 behavior (backward compat)
- Add `X-Selected-Node` response header with peer_id

**Done criteria:**
- With forwarder: request forwarded to selected node
- Without forwarder: 501 returned (backward compat)
- 502 on forwarding error, 504 on timeout
- X-Selected-Node header present
