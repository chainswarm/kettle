| D004 | M005/S01/T01 | architecture | Whether LOCAL_RPC and DEV_RPC are module constants in subnet/hypertensor/config.py or env var names resolved at runtime | They are env var names only — resolved via os.environ.get() in subnet/cli/run_node.py, not constants in config.py | subnet/hypertensor/config.py contains only BLOCK_SECS, EPOCH_LENGTH, and SECONDS_PER_EPOCH. Hypertensor already owns this concern; the wrapper's job is serialisation (asdict) and error normalisation (None on exception, receipt on is_success=False). | D006 | M005/S02/T01 | architecture | Whether ChainScoreSubmitter.submit() should short-circuit on an empty score list | No short-circuit — empty list passes data=[] to propose_attestation unchanged | The chain accepts empty submissions as valid. Short-circuiting silently changes observable behaviour (no extrinsic broadcast vs. empty broadcast) and may mask bugs in the caller. Passing through preserves predictable, testable behaviour at the cost of one no-op extrinsic. | Yes — if the chain rejects empty submissions with a specific error, add an explicit early-return with a logged warning |
| D007 | M005/S02/T02 | architecture | How docker-compose.chain.yml handles the PHRASE anchor value vs per-service overrides for signing nodes | x-chain-env anchor PHRASE set to "" (empty literal); validator, miner-1, miner-2 each add PHRASE: ${SERVICE_PHRASE:?...} as a per-service override | Using ${PHRASE:-} in the anchor would silently pick up any ambient PHRASE env var if set in the shell, causing the bootnode (or any service without an explicit override) to use an unintended mnemonic. Setting the anchor to "" means bootnode always gets empty (safe for read-only chain queries). Per-service :? guards fail fast with a descriptive error when the operator forgets to supply a hotkey mnemonic, preventing silent misconfiguration. | Yes — if bootnode is ever given extrinsic-signing duties, it should get its own :? guard |
| D008 | M005/S03 | architecture | Constructor signature for ChainOverwatchReporter — two IDs needed vs. boundary map's one | Constructor takes (hypertensor, overwatch_node_id, subnet_id) — three args including subnet_id | The boundary map specifies (hypertensor, overwatch_node_id) but commit_overwatch_subnet_weights requires commit_weights=[{subnet_id: N, weight: bytes}] — subnet_id is always required to build the commit list. Without it the constructor would need subnet_id passed on every slash() call, breaking the thin-wrapper contract where all fixed config is injected at construction time. subnet_id is always known at instantiation (same as ChainScoreSubmitter). | Yes — if overwatch nodes ever submit weights for multiple subnets simultaneously, pass a list of subnet_ids instead |
| D009 | M005/S03 | architecture | How _overwatch_epoch_loop conditionally instantiates ChainOverwatchReporter without breaking MOCK_TEE mode | Reporter guarded behind OVERWATCH_NODE_ID env var; None when unset; MOCK_TEE mode unaffected | _overwatch_epoch_loop has no concept of "chain mode" vs "local mode" — it receives hypertensor but that could be LocalMockHypertensor in MOCK_TEE mode. Gating on OVERWATCH_NODE_ID (an env var that only exists in chain-connected deployments) is the simplest, non-invasive guard: when OVERWATCH_NODE_ID is unset, reporter=None and slash() is never called. This avoids adding a new parameter to _overwatch_epoch_loop and keeps the existing call site in server.py unchanged. | Yes — if overwatch node ID is later surfaced as a server attribute, pass it as a constructor arg instead |
| D010 | M005/S03 | architecture | Which docker-compose.chain.yml service gets the OVERWATCH_PHRASE credential guard | OVERWATCH_PHRASE added to validator service (not a separate overwatch service) since _overwatch_epoch_loop runs inside the validator node | _overwatch_epoch_loop is started in the same trio nursery as the validator's other loops (server.py line ~396). The validator node is the process that instantiates ChainOverwatchReporter and calls commit/reveal extrinsics. No separate overwatch service exists — adding a dedicated service would require a new container entry, new key file, and new CLI entrypoint, which is S04 scope. For S03, the validator service gets OVERWATCH_PHRASE and OVERWATCH_NODE_ID, consistent with how VALIDATOR_PHRASE already covers propose_attestation signing in that same process. | Yes — a dedicated overwatch service with its own container is a valid future topology |
| D011 | M005/S04/T02 | architecture | Whether the CI Layer 3 chain smoke test step should fail the build when no testnet is available | continue-on-error: true — Layer 3 step is informational only; CI never fails on expected chain absence | No testnet node is available in CI. | D012 | M005/S04/T02 | observability | How to distinguish "chain is reachable but no scores yet" from "chain is unreachable" in check scripts and documentation | [WARN] means no data yet (first 2-3 epochs); ERROR: means connection failure — documented in CHAIN.md and distinguishable in all check scripts | New developers consistently misread [WARN] output as a failure when the subnet has just started. This distinction is documented in CHAIN.md's [WARN] vs [OK] semantics table and in TESTING_LAYERS.md. All four check scripts (check_peers.py, check_scores.py, check_slash.py, smoke_test_chain.py) follow this convention. | No — changing this convention would require updating all four scripts and their documentation simultaneously |
| D013 | M005/S04/T01 | architecture | How to accumulate and submit ChainScoreSubmitter scores across a multi-peer epoch loop in _validator_scoring_loop | scores[] reset to [] at the top of each epoch block; only peers that complete score_peer() without raising are appended; submit() called once after the per-node for loop | Resetting per epoch prevents stale scores from a crashed epoch leaking into the next.
